# TLS Proxy Modernization Requirements

## Objective

Modernize the Rust proxy and its management UI while preserving the existing
TLS passthrough behavior and single-binary distribution model.

The completed application must support two listener modes:

1. TLS passthrough: inspect the ClientHello SNI and forward the original,
   encrypted TLS stream to the upstream server.
2. TLS termination: accept and decrypt TLS from the client, then forward
   plaintext TCP traffic to a non-TLS upstream server.

The admin server must support certificates automatically generated and signed
by a managed local CA for easy secure startup. Admin certificates and
proxy-listener certificates are separate concerns and must not be reused
implicitly.

## General Constraints

- Preserve backward compatibility with existing `config.yaml` files whenever
  practical.
- Existing listeners without a mode setting must continue to use TLS
  passthrough.
- The release binary must embed the complete production UI.
- The production UI must not require a CDN, Google Fonts, or any other runtime
  Internet resource. Fonts, icons, scripts, styles, and images must be embedded
  or replaced with local/system equivalents.
- Do not commit generated private keys, certificates, build artifacts, or local
  test output.
- Add and commit `Cargo.lock`.
- Configuration errors must produce actionable messages and must not panic.
- Private keys must never be returned by an API or written to logs.
- Every leaf certificate automatically generated by this application, for the
  admin server or any future feature, must be signed by the configured local CA
  described in section 4.

## 1. Dependency And Toolchain Upgrade

Review every Rust and UI dependency and update it to a currently supported,
stable version where compatible with the project.

### Rust requirements

- Add a minimum supported Rust version to `Cargo.toml`.
- Generate and commit `Cargo.lock`.
- Remove dependencies that become unused.
- Avoid duplicate functionality where the standard library or an existing
  dependency is sufficient.
- Replace deprecated APIs and resolve compiler and Clippy warnings introduced
  by the upgrade.
- Do not blindly update dependencies when a newer version would require an
  unjustified rewrite. Document any dependency deliberately left behind.

### UI requirements

- Replace the unsupported Angular application with a dependency-free embedded
  web console using semantic HTML, CSS, and browser-native JavaScript modules.
- Preserve all existing admin capabilities.
- Do not require Node.js or a frontend package manager to build a release.

### Acceptance criteria

- `cargo build --locked --release` succeeds.
- `cargo test --locked` succeeds.
- `cargo clippy --locked --all-targets -- -D warnings` succeeds, except for
  explicitly documented and narrowly allowed lints.
- The embedded UI passes JavaScript syntax and external-resource checks.
- A clean checkout can reproduce the build using documented tool versions.

## 2. Responsive Embedded Management UI

Redesign the management UI as a professional, responsive application.

### Required screens and behavior

- Dashboard:
  - manager running/stopped state;
  - listener health;
  - active and total connections;
  - uploaded and downloaded byte totals;
  - failed listeners and their error messages;
  - last refresh time and manual refresh.
- Listener configuration:
  - create, edit, and delete listeners;
  - configure bind address, target port, ACL policy, static hosts, regex
    patterns, idle timeout, speed limit, and listener mode;
  - show termination certificate fields only when termination mode is selected;
  - validate required fields before saving.
- DNS overrides:
  - create, edit, and delete exact, suffix, and regex overrides;
  - explain matching priority;
  - validate regex expressions and target host/port formats.
- Admin TLS settings:
  - configure explicit certificate paths or automatic certificate generation;
  - configure generated-certificate SAN hostnames and IP addresses;
  - explain that generated local-CA-signed certificates are not publicly
    trusted until clients trust the local CA.
- Service controls:
  - save without applying;
  - apply/restart;
  - start;
  - stop;
  - restore the last applied configuration;
  - require confirmation for disruptive operations.

### UX requirements

- Work on desktop, tablet, and mobile widths.
- Do not rely on wide editable tables for the primary mobile experience.
- Provide visible loading, success, validation, empty, and error states.
- Use accessible labels, keyboard navigation, sufficient contrast, and visible
  focus states.
- Format byte counts and durations for humans while preserving exact values
  where useful.
- Avoid long-lived snackbars for routine success notifications.

### Embedding requirements

- Build the production UI before compiling the Rust release binary.
- Embed the resulting assets using the existing embedded-static-files approach
  or an equivalent compile-time mechanism.
- Remove all runtime external resource requests.
- The UI must load and remain usable while the host has no Internet access.

## 3. Standards-Based ClientHello And SNI Inspection

Replace the handwritten TLS ClientHello parser with a maintained TLS protocol
implementation, preferably `rustls::server::Acceptor`.

### Passthrough behavior

- Read the ClientHello incrementally; do not assume it fits in one TCP read or
  one TLS record.
- Extract SNI using the library API.
- Preserve every byte read from the client.
- After ACL and routing checks succeed, write the preserved bytes to the
  upstream exactly once and then continue bidirectional passthrough.
- Do not complete or terminate the TLS handshake in passthrough mode.

### Limits and errors

- Retain a configurable or clearly defined ClientHello read timeout.
- Enforce a documented maximum buffered handshake size to prevent unbounded
  memory use.
- Reject missing SNI unless a future explicit fallback route is configured.
- Reject malformed TLS with a clear log message and close the connection.
- Treat normal client protocol errors as connection failures, not process
  failures.
- Extract ALPN only if useful for diagnostics or future routing; it is not a
  routing requirement in this task.

### Acceptance criteria

- Fragmented ClientHello messages are parsed successfully.
- ClientHello messages larger than the old 4096-byte buffer work up to the new
  configured limit.
- TLS passthrough remains end-to-end encrypted between client and upstream.
- The upstream receives the original ClientHello without duplication or loss.

## 4. Automatic Local CA And TLS For The Admin Server

Add automatic local certificate-authority management and certificate generation
for the Rocket admin server. This feature is intended to provide encrypted
first-run access without requiring the operator to run OpenSSL manually. It does
not provide public trust until clients explicitly trust the generated CA.

### Configuration

Support an admin TLS configuration equivalent to:

```yaml
admin_server:
  bind_address: 0.0.0.0
  bind_port: 48888
  username: admin
  password: change-me
  tls: true
  tls_cert: null
  tls_key: null
  tls_ca_cert: null
  mutual_tls: false
  auto_generate_tls: true
  certificate_ca_path: ./CA.pem
  certificate_ca_key_path: ./CA.key
  generated_cert_path: admin-server.pem
  generated_key_path: admin-server.key
  certificate_hostnames:
    - localhost
    - proxy.example.internal
  certificate_ip_addresses:
    - 127.0.0.1
    - "::1"
  certificate_validity_days: 365
```

Exact Rust field names may vary, but serialized YAML names and semantics must be
clear and documented.

`tls_ca_cert` retains its existing meaning as the trust anchor used to verify
client certificates when admin mTLS is enabled. It is not the CA used to sign
automatically generated server certificates. `certificate_ca_path` and
`certificate_ca_key_path` identify the signing CA.

### Local CA management

- All certificates generated by this application must be leaf certificates
  signed by a persisted local CA. Do not generate unrelated self-signed leaf
  certificates.
- If no CA certificate path is configured, default to `./CA.pem`.
- If no CA private-key path is configured, default to `./CA.key`.
- Resolve relative paths against the application's working directory, matching
  the existing configuration-file behavior.
- If neither CA file exists, generate a new CA certificate and private key and
  write them to the configured/default paths.
- The generated CA certificate must:
  - be marked as a CA certificate;
  - contain appropriate CA key usages;
  - use a cryptographically secure key and signature algorithm supported by
    current rustls clients;
  - have a documented subject name and validity period.
- If both CA files exist, load and validate them before generating any leaf
  certificate.
- If only one of the CA certificate or CA key exists, fail with an actionable
  error. Do not replace the existing file or silently create a new unmatched
  pair.
- If the files exist but are unreadable, malformed, expired, not a CA, or the
  private key does not match the certificate, fail with an actionable error.
- Never regenerate or rotate an existing valid CA automatically. Changing the
  CA would invalidate client trust. CA rotation must be an explicit operator
  action.
- Write newly generated CA files atomically.
- On Unix, create `CA.key` with owner-only permissions where possible.
- Never return the CA private key through an API or write it to logs.
- The UI and documentation must provide the CA certificate for operator
  download or clearly identify its path so clients can install it as a trusted
  local CA. The CA private key must never be downloadable.

### Precedence and behavior

- If `tls` is false, run HTTP and do not generate certificates.
- If `tls` is true and both explicit `tls_cert` and `tls_key` are configured,
  use them and do not generate replacements.
- If `tls` is true, explicit files are not configured, and
  `auto_generate_tls` is true:
  - load or create the configured/default local CA;
  - generate or reuse the configured admin leaf certificate and private key;
  - ensure the leaf certificate is signed by that CA.
- If TLS is requested but neither a complete explicit pair nor automatic
  generation is available, fail startup with a clear error.
- Generate the certificate before Rocket configuration is constructed.
- Include configured DNS names and IP addresses as SAN entries.
- Do not use arbitrary request Host headers or observed client SNI values to
  expand a certificate.
- Reuse an existing valid generated certificate when its key matches, its SANs
  satisfy the configured names, it chains to the currently configured CA, and
  it is not near expiry.
- Regenerate when files are absent, invalid, mismatched, missing required SANs,
  signed by a different CA, or within a documented renewal window.
- Write certificate and key files atomically.
- On Unix, create private-key files with owner-only permissions where possible.
- Never overwrite explicitly configured certificate files.
- Automatic mTLS CA/client-certificate generation is out of scope unless added
  as a separately documented feature.

### UI and documentation

- Expose these options in the admin settings UI.
- Clearly state that browsers and clients will warn until the generated
  `CA.pem` is installed as a trusted local CA.
- Continue to support explicit certificates for operators using an internal CA
  or publicly trusted CA.

## 5. Listener TLS Termination To Plaintext Upstream

Add a listener-level mode for accepting TLS from clients and forwarding
decrypted plaintext to a non-TLS upstream.

This is not merely an "upstream no TLS" transport toggle. The proxy must own the
inbound TLS endpoint and therefore must have a certificate and private key that
the client accepts.

### Configuration

Use an explicit listener mode rather than an ambiguous boolean:

```yaml
listeners:
  passthrough-example:
    bind: 0.0.0.0:443
    mode: passthrough
    target_port: 443
    policy: DENY
    rules:
      static_hosts: []
      patterns: []
    max_idle_time_ms: 3600000
    speed_limit: 0.0

  termination-example:
    bind: 0.0.0.0:8443
    mode: terminate
    target_port: 8080
    tls_cert: certs/example.pem
    tls_key: certs/example.key
    policy: ALLOW
    rules:
      static_hosts:
        - example.com
      patterns: []
    max_idle_time_ms: 3600000
    speed_limit: 0.0
```

### Mode semantics

- `mode` is optional for backward compatibility and defaults to `passthrough`.
- `passthrough`:
  - inspect SNI;
  - apply ACL and DNS override routing;
  - connect to the upstream;
  - replay the ClientHello and pass encrypted bytes unchanged;
  - reject listener-level termination certificate fields as unused or warn
    clearly.
- `terminate`:
  - require readable `tls_cert` and `tls_key` files at listener startup;
  - accept the inbound TLS handshake using rustls/tokio-rustls;
  - obtain and validate SNI before allowing application data forwarding;
  - apply the same ACL, DNS override, local-address protection, timeout, speed
    limiting, statistics, active tracking, and cancellation behavior as
    passthrough mode;
  - connect to the resolved upstream using ordinary plaintext `TcpStream`;
  - relay decrypted client application bytes to the upstream and plaintext
    upstream responses through the encrypted client connection.

### Certificate rules

- Item 4's admin certificate is only for the admin server and must not be
  silently reused by listeners.
- Listener termination initially supports one explicitly configured
  certificate/key pair per listener.
- The certificate may contain multiple SANs.
- The configured ACL should normally restrict SNI names to names covered by the
  listener certificate.
- If the client sends an SNI not covered by the configured certificate, the TLS
  library may complete with a certificate-name failure at the client. The
  server must still apply its ACL and must not route an unauthorized name.
- Dynamic per-SNI certificate maps, ACME issuance, TLS interception using a
  generated CA, and arbitrary on-demand certificates are out of scope for this
  requirement.

### Protocol scope

- This is generic TLS-to-plaintext TCP forwarding, not an HTTP reverse proxy.
- Do not parse, rewrite, or inject HTTP headers.
- Preserve half-close and shutdown behavior where practical.
- Existing speed limits continue to represent a shared per-connection limit
  across both directions unless explicitly changed and documented.
- Client-to-proxy traffic remains encrypted; proxy-to-upstream traffic is
  plaintext by design and must be documented as such.

### Validation

- A terminating listener without both certificate and key must fail to start
  without preventing unrelated valid listeners from starting.
- Invalid certificate/key pairs must produce an actionable listener status.
- Configuration deserialization must continue to accept old listeners with no
  mode or TLS fields.
- The UI must explain the security consequences of plaintext upstream traffic.

## 6. Automated Tests And Upgrade Verification

The existing `test/*.sh` files are manual examples with hardcoded addresses;
they are not sufficient as the regression suite. Replace or supplement them
with self-contained automated tests containing assertions.

### Required Rust tests

- Configuration:
  - old configuration compatibility;
  - listener mode defaults;
  - termination certificate validation;
  - admin certificate-generation precedence.
- ACL:
  - ALLOW and DENY policies;
  - static hostname case behavior;
  - regex behavior.
- DNS overrides:
  - exact match priority;
  - longest suffix priority;
  - deterministic regex priority;
  - host and port parsing.
- ClientHello:
  - normal SNI;
  - fragmented reads;
  - multiple TLS records where supported;
  - missing SNI;
  - malformed input;
  - oversized input;
  - timeout.
- Admin certificates:
  - default CA paths;
  - initial CA generation;
  - reuse of an existing CA;
  - rejection of a partial or mismatched CA pair;
  - no automatic replacement of an invalid existing CA;
  - initial generation;
  - SAN contents;
  - leaf certificate signature and chain to the configured CA;
  - reuse;
  - regeneration conditions;
  - explicit-file precedence;
  - invalid configuration.
- Proxy integration:
  - TLS passthrough to a local TLS test server;
  - correct SNI routing;
  - denied SNI;
  - TLS termination to a local plaintext echo or HTTP server;
  - byte counters;
  - idle timeout;
  - speed limiting where reliable;
  - stop/restart cancellation.
- Admin API:
  - authentication;
  - configuration read/write;
  - start, stop, apply, and reset;
  - listener errors returned without crashing the service.

All network integration tests must bind to loopback with dynamically assigned
ports. They must not depend on a specific LAN address, external DNS, public
Internet access, or pre-existing certificate files.

### Required UI tests

- Production build succeeds.
- Listener forms validate mode-specific fields.
- Admin automatic-certificate settings serialize correctly.
- Existing configuration, DNS, statistics, and service-control workflows work.
- Key responsive layouts have component or end-to-end coverage.
- A build-time check rejects external runtime URLs in generated UI assets.

### Final verification

Run and report:

```text
cargo fmt --check
cargo test --locked
cargo clippy --locked --all-targets -- -D warnings
cargo build --locked --release
Embedded JavaScript syntax check
Embedded UI external-resource check
```

Also run local end-to-end smoke tests for both listener modes and verify that a
release binary serves its UI successfully with outbound Internet access
disabled.

## Recommended Implementation Order

1. Add baseline automated tests and reproducible lockfiles.
2. Upgrade Rust dependencies and toolchain metadata.
3. Replace handwritten ClientHello parsing while preserving passthrough.
4. Implement local CA management and admin leaf-certificate generation.
5. Introduce listener modes and TLS termination.
6. Upgrade and redesign the UI, including all new settings.
7. Run the complete regression, release, and offline-UI verification suite.

## Definition Of Done

The work is complete only when:

- existing passthrough configurations continue to work;
- malformed or unsupported connections cannot crash the process;
- the admin server can securely bootstrap with a generated local CA and a leaf
  certificate signed by that CA;
- terminating listeners decrypt client TLS and forward plaintext upstream using
  explicitly configured listener certificates;
- the responsive UI exposes and validates all supported settings;
- the binary embeds all UI assets and works without runtime Internet access;
- automated tests cover both listener modes and certificate generation; and
- all documented build and verification commands pass.
